Load or Create Tenant

No prerequisites

From the home page, select an existing tenant or click Create New Tenant. A new tenant needs these identity fields:

FieldReqWhat it does
tenant_idRequiredUnique slug (lowercase, underscores). Cannot change after creation.
chat_titleRequiredDisplayed in the widget header. Max 100 chars.
welcome_messageRequiredFirst message the user sees. Max 500 chars.
tone_promptRequiredPersonality instructions for the AI. Max 2000 chars. This is one of the most important fields — it shapes every response.
subscription_tierRequiredFree / Standard / Premium / Enterprise. Controls feature availability.

AWS Configuration

Navigate to Settings > AI & AWS tab:

FieldReqWhat it does

aws_regionRequiredRegion where the KB lives (e.g., us-east-1).
When you're done
You have a loadable tenant with identity and AWS connectivity. Nothing else exists yet.

Programs

Prerequisite: Tenant loaded

Programs are organizational containers. Every form must belong to a program. If the client has one product or service, one program is fine. If they serve multiple audiences or have distinct offerings, create a program for each.

FieldReqValidationWhat it does
program_idRequiredLowercase, numbers, underscores. Max 50.Unique key. Used by forms, CTAs, and branches to group entities.
program_nameRequiredMax 100 charsDisplay name (shown in builder UI, not to end users).
descriptionOptionalMax 500 charsInternal notes about what this program covers.
Example
A nonprofit with mentoring and food distribution: dare2dream ("Dare2Dream Mentoring Program") and lovebox ("Love Box Distribution Program").
When you're done
You have at least one program. You can now create forms.

CTAs (Call-to-Actions)

Prerequisite: Forms exist (if any CTA uses start_form)

CTAs are the buttons users click. They're the core routing mechanism — every branch, action chip, and showcase card ultimately surfaces CTAs.

CTA Fields

FieldReqValidationWhat it does
ctaIdRequiredUnique. Letters, numbers, hyphens, underscores.Internal key. Branches and chips reference this.
labelRequiredMax 100 charsButton text the user sees (e.g., "Apply to Mentor").
actionRequiredEnum (see below)What happens when the button is clicked.
typeRequiredAuto-set from actionInternal classification. Don't set manually.
ai_availableOptionalBooleanWhen true, the AI can suggest this CTA dynamically (Tier 1-2 scoring).
target_branchOptionalMust reference existing branchNavigates to a conversation branch when clicked.
on_completion_branchOptionalMust reference existing branchBranch to show after a form completes.
program_idOptionalMust reference existing programOrganizational grouping.

Action Types

ActionType (auto)Required fieldWhat happens
start_formform_triggerformId (must reference existing form)Opens the specified conversational form.
external_linkexternal_linkurl (valid HTTP/HTTPS)Opens URL in new tab.
send_querybedrock_queryquery (text)Sends query to the AI as if the user typed it.
show_infoinfo_requestprompt (text)Displays a static info message (no AI call).

CTA Design Patterns

Form trigger with follow-up branch
Action: start_form, formId: mentor_application
on_completion_branch: mentor_next_steps
After the form submits, the user sees the next-steps branch CTAs.
Navigation CTA (rigid path)
Action: show_info, prompt: "Here's what you need to know..."
target_branch: dare2dream
Clicking routes to the branch, which shows its own CTAs. The AI exits the conversation loop.
AI-available CTA (flexible path)
Action: send_query, query: "Tell me about volunteering"
ai_available: true — No target_branch
The AI stays in the loop and may suggest follow-up chips.
When you're done
You have CTAs for all user actions. You can now group them into branches.
5

Branches

Prerequisite: CTAs exist

Branches group CTAs into routing destinations. When the AI (or a click) routes to a branch, the branch's primary and secondary CTAs are displayed as buttons.

FieldReqValidationWhat it does
branchIdRequiredUnique identifierKey used by CTAs (target_branch), chips, and intent definitions.
available_ctas.primaryRequiredMust reference existing CTA; not in secondaryThe featured/prominent CTA button.
available_ctas.secondaryRequiredEach must reference existing CTA; no duplicatesAdditional CTA buttons shown below the primary.
descriptionOptionalFree textWhen should this branch activate? Useful for intent definitions.
program_idOptionalMust reference existing programOrganizational grouping.
showcase_item_idOptionalMust reference existing showcase itemDisplay a content card alongside the branch CTAs.

How Branches Work at Runtime

  1. User clicks a CTA with target_branch: "dare2dream" (or AI routes there)
  2. Lambda looks up conversation_branches.dare2dream
  3. Resolves available_ctas.primary → full CTA object from cta_definitions
  4. Resolves available_ctas.secondary → array of full CTA objects
  5. Sends all resolved CTAs to the widget as buttons
Branch design tip
The primary CTA should be the most important action for that path. Secondary CTAs are supporting options. Example: Branch dare2dream — primary = "Apply to Be a Mentor", secondary = ["Learn More About Mentoring", "Back to Programs"].
When you're done
You have branches that organize CTAs into paths. The core routing structure is complete. Steps 6–7 are optional enhancements.

Validate & Deploy

Prerequisite: All previous steps complete

Validation

Check the validation summary on the Settings page or the Dashboard for status indicators.

Errors (must fix)Warnings (should fix)
  • Form references a program that doesn't exist
  • CTA references a form that doesn't exist
  • Branch references a CTA that doesn't exist
  • Missing required fields
  • Missing fallback_branch when intent definitions are configured
  • Orphaned entities (CTA not used by any branch)
  • conversational_forms disabled but forms exist
  • Intent definition references missing branch

Deploy

  1. Click Deploy to S3 in the header
  2. Review the deployment summary (entity counts, version bump)
  3. Confirm
  4. The builder merges your edits, creates a backup, increments the version by 0.1, and uploads to S3
  5. The Lambda picks up the new config on next request (cache TTL: 5 minutes)
Done
Your tenant is live. Test the widget embed to verify the configuration.
AI

Intent Definitions

Prerequisite: Branches configured, fallback_branch set

Intent definitions power the classification pipeline. After each AI response, a separate classifier reads the user's message and matches it against these definitions to determine which branch CTAs to display. Navigate to Intent Definitions (available from the sidebar or Settings > AI & AWS tab).

"intent_definitions": [
  {
    "name": "mentoring_applicant",
    "description": "The visitor wants to BECOME a mentor -- asking how to apply, requirements, time commitment. NOT when someone wants to RECEIVE mentorship.",
    "target_branch": "dare2dream"
  },
  {
    "name": "donating",
    "description": "User wants to donate money, items, or gifts.",
    "target_branch": "donate"
  }
]
FieldReqValidationWhat it does
nameRequiredLowercase + underscores, max 100, uniqueIntent identifier. Logged in CloudWatch.
descriptionRequiredMin 10, max 1000 charsWhat the classifier reads. Quality determines accuracy.
target_branchOptionalMust reference existing branchBranch to route to when this intent matches.
cta_idOptionalMust reference existing CTASingle CTA to surface (if no branch).
Description quality tips
  • Include what the intent IS and what it IS NOT
  • Use the user's perspective ("The visitor wants to...")
  • Differentiate similar intents explicitly (mentor applicant vs. mentee)
  • Keep under ~200 chars for best classifier performance

Routing priority: target_branch > cta_id > fallback. If an intent matches but has neither, the fallback branch is used. Max 12 intent definitions per tenant.

Common Patterns

Simple Lead Capture

  1. One program (e.g., general)
  2. One form: name, email, phone
  3. One CTA: start_form → the form
  4. One branch with that CTA as primary
  5. Set as fallback_branch

Multi-Program Nonprofit

  1. Program per initiative
  2. Forms per program (volunteer, donate, enroll)
  3. CTAs: form triggers + info CTAs
  4. Branches per program path
  5. Action chips for top-level nav
  6. Intent definitions to classify into the right program

Information-Only (No Forms)

  1. One program
  2. No forms
  3. CTAs: show_info or send_query
  4. Branches grouping related info CTAs
  5. Action chips for quick navigation
  6. conversational_forms feature OFF
